# Frankenstein – Full Modular, Redundant, Multi-Chat System Blueprint

## 0. Goal

Build a single Android app (“Frankenstein”) that is:

- Modular, multi-layer, and fault-tolerant.
- A hub for:
  - Multi-provider chat (ChatMultiAPI).
  - Embedded code editor.
  - Embedded Termux shell.
  - Embedded browser tab.
  - Password manager.
  - VoIP (embedded + standalone FreedomVOIP).
  - Identity / privacy / sandbox / mesh layers.
- Backed by a Python orchestration backend.
- Wired with redundancy meshes (heartbeat, logging, alarm).
- Designed to be extended and regenerated by an AI workflow.

## 2. IdentityLayer

Purpose: per-device, per-app identity with no global leakage.

Components:

- `IdentityManager`
  - Generates and manages:
    - `family_id`
    - `node_id`
  - Uses local key generation (`keygen`) and storage (`keystore`).
- `IdentityConfig`
  - Flags:
    - participation enabled/disabled
    - privacy level
    - mesh participation modes
- Storage:
  - Encrypted local file(s).
  - Optional redundant backup file for recovery.

Rules:

- Identity never leaves device in raw form.
- Identity used to tag logs/mesh events in a privacy-safe way.

---

## 3. PrivacyTunnel

Purpose: ensure sensitive operations stay local or follow strict rules.

Components:

- `TunnelInit`
  - Initializes privacy tunnel context.
- `TunnelRoute`
  - Decides routing:
    - local-only
    - remote provider
    - blocked
- `BrowserProxy`
  - Routes browser tab traffic through tunnel when needed.
- `ProviderPrivacyRules`
  - Per-provider privacy policies (what can be sent, what must stay local).

Rules:

- If tunnel fails, operations are blocked rather than silently going remote.
- Logging mesh records tunnel decisions and failures.

---

## 4. Sandbox (Termux integration)

Purpose: isolated environment for commands, scripts, backend tasks.

Components:

- `SandboxManager`
  - Starts/stops Termux sessions.
  - Manages lifecycle.
- `EnvSnapshot` / `EnvRestore`
  - Capture environment state (files, configs).
  - Restore to previous state.
- `SandboxRunner`
  - Runs commands/scripts in Termux.
  - Streams output to UI.

Termux tab:

- Embedded terminal view.
- Connected to SandboxManager.
- Shows live output from sandbox.

Rules:

- Sandbox failures are logged to logging mesh.
- Snapshot/restore used for resilience after crashes.

---

## 5. NetworkParticipation

Purpose: control whether app participates in meshes and networked features.

Components:

- `ParticipationConfig`
  - Flags:
    - heartbeat mesh on/off
    - logging mesh on/off
    - alarm mesh on/off
    - remote provider usage on/off
- `IdleOnlyMode`
  - Only participate when device is idle.

Rules:

- Meshes degrade gracefully if participation is disabled.
- UI shows participation status in Security tab.

---

## 6. DistributedBuild

Purpose: orchestrate multi-step tasks (builds, workflows, backend jobs).

Components:

- `BuildPlanner`
  - Breaks tasks into steps.
- `BuildOrchestrator`
  - Executes steps, handles dependencies.
- `ArtifactStore`
  - Stores intermediate results and final artifacts.

Use cases:

- Building VoIP binaries.
- Running backend workflows.
- Complex multi-provider operations.

Rules:

- Tasks can be retried.
- Artifacts stored locally for recovery and reuse.

---

## 7. WiringAtlas (Redundant wiring)

Purpose: central registry of all modules, routes, providers, meshes, tabs.

Components:

- `WiringAtlas`
  - Knows:
    - modules (core, chat, ui, voip, backend, sandbox)
    - routes (actions → modules)
    - providers (chat, voip, backend)
    - meshes (heartbeat, logging, alarm)
    - tabs (UI entry points)
- `RouteMap`
  - Maps actions to primary and backup routes.

Redundancy:

- Critical actions have:
  - primary route
  - backup route(s)
- Provider routing uses WiringAtlas to find:
  - primary provider
  - backup providers
  - local fallback (e.g. local LLM)

---

## 8. Multi-Chat System (ChatMultiAPI)

Purpose: multi-provider chat with redundancy and failover.

Components:

- `Router.kt`
  - Entry point for chat requests.
  - Uses WiringAtlas + ProviderRouter.
- `ProviderRouter.kt`
  - Provider-specific routing logic.
- `ProviderRegistry`
  - List of providers:
    - Grok
    - local LLM
    - remote APIs
    - custom providers
- `FailoverManager`
  - Handles:
    - provider failures
    - retries
    - fallback to backup providers
    - fallback to local provider.

Multi-chat UI:

- **Chat Hub tab**
  - Combined view of conversations across providers.
  - Filter by provider, status, tags.
- **Per-provider chat tabs**
  - Each provider gets its own tab/view.
  - Uses shared ChatMultiAPI core.

Rules:

- If a provider fails, FailoverManager chooses backup provider.
- Logging mesh records provider failures and failover events.

---

## 9. VoIP – FreedomVOIP

### Embedded FreedomVOIP (inside Frankenstein)

Components:

- `VoipEngine`
  - SIP/WebRTC core.
- `VoipConfig`
  - server, credentials, codecs.
- `VoipController`
  - start/stop calls, status, events.

Integration:

- Accessible from:
  - Chat tab (call button).
  - VoIP tab (if present).
  - Settings/VoIP config.

Redundancy:

- Logs routed to logging mesh.
- Failures trigger alarm mesh events.

### Standalone FreedomVOIP module

Components:

- `FreedomVoipActivity`
  - main UI for standalone app.
- `VoipServerBootstrap`
  - starts local VoIP server (e.g. Asterisk/FreeSWITCH).
- `VoipSettingsActivity`
  - config UI.

Rules:

- Shares VoipEngine with embedded module.
- Built as separate APK.

## 10. Tabs and UI

Tabs:

1. **Chat Hub tab**
   - Multi-provider chat overview.
   - Uses ChatMultiAPI.

2. **Provider chat tabs**
   - One per provider.
   - Direct access to specific provider.

3. **Code Editor tab**
   - File browser for local project files.
   - Editor with:
     - syntax highlighting
     - save
     - basic search
   - Works with backend and sandbox.

4. **Termux tab**
   - Embedded terminal view.
   - Connected to SandboxManager.
   - Shows live Termux output.

5. **Browser tab**
   - WebView-based.
   - Opens:
     - local docs
     - dashboards
     - safe external URLs
   - Uses PrivacyTunnel rules.

6. **Password Manager tab**
   - Local encrypted credential store.
   - Components:
     - `PasswordStore`
     - `PasswordEntry`
     - `PasswordManagerActivity` / fragment.
   - No cloud sync.

7. **Settings tab**
   - Toggles:
     - providers
     - meshes
     - redundancy options
     - privacy options
     - VoIP settings
     - Termux integration
     - backend options.

8. **Security tab**
   - Shows:
     - identity summary (non-sensitive)
     - mesh status (heartbeat/logging/alarm)
     - privacy tunnel status
     - redundancy status (providers, routes).

9. **Logs/Crash tab**
   - Crash logger:
     - `CrashLogger`
     - `CrashLogActivity`
   - General logs:
     - `LogStore`
   - Integrated with logging mesh.

10. (Optional) **Projects/API Keys tabs**
    - Projects: manage local projects, configs.
    - API Keys: manage provider keys (local storage).

## 11. Meshes

### Heartbeat mesh

- Periodic “I’m alive” signals.
- Configurable:
  - local-only
  - networked (if allowed).

### Logging mesh

- Structured logs for:
  - provider events
  - tunnel events
  - sandbox events
  - VoIP events
  - backend events
  - UI events
  - crashes.

### Alarm mesh

- Alerts for:
  - crashes
  - security events
  - VoIP failures
  - backend failures
  - redundancy failures (no providers available)
  - tunnel failures.

Rules:

- Mesh participation controlled by NetworkParticipation.
- Mesh status visible in Security tab.

## 15. Generation rules (for AI workflow)

When generating code and structure from this blueprint:

1. **Do NOT depend on legacy two-layer Android structure.**
2. Create a clean multi-module Gradle project as described.
3. Implement:
   - IdentityLayer
   - PrivacyTunnel
   - Sandbox
   - NetworkParticipation
   - DistributedBuild
   - WiringAtlas
   - Meshes (heartbeat, logging, alarm)
   - ChatMultiAPI core (Router, ProviderRouter, ProviderRegistry, FailoverManager)
   - Embedded FreedomVOIP
   - Standalone FreedomVOIP module
   - All tabs and UI components listed
   - Python backend modules
   - Blueprints/docs files
4. Implement redundancy:
   - multiple routes for critical actions
   - provider failover
   - local fallback where possible.
5. Use Kotlin for Android code, Python for backend.
6. Ensure the project can build with:
   - `./gradlew assembleRelease`
7. Keep code readable and reasonably simple, but respect modular boundaries.
